# TDD 流程检核清单

**版本**: v1.0

## 🎯 核心原则

**TDD = Tests First, Always!**

```
测试代码不存在 → DEV 禁止开始实现
测试未失败（RED）→ DEV 禁止开始实现
测试未全绿（GREEN）→ 代码禁止提交
```

---

## 📋 TDD 完整流程检查点

### 🔴 检查点 1: PO 完成需求定义

**责任人**: PO
**触发条件**: 开始新功能/Week 开发

**门禁标准**:
- [ ] `docs/week-N-WO.md` 已创建
- [ ] `docs/week-N-PL.md` 已创建
- [ ] 验收标准明确定义
- [ ] PM 已评审并批准

**验证命令**:
```bash
# 检查文档存在
ls docs/week-N-WO.md docs/week-N-PL.md
```

**输出产物**: WO 文档 + PL 文档

**PM 签认**: ✅ ___________ (日期)

---

### 🔴 检查点 2: QA 编写测试代码（RED）

**责任人**: QA
**前置条件**: 检查点 1 通过
**触发条件**: PO 完成需求定义

**门禁标准**:
- [ ] 测试代码文件已创建（`src/__tests__/week-N/*.test.ts`）
- [ ] 测试用例已实现（非文档，是实际代码）
- [ ] 运行测试显示失败（RED 状态）
- [ ] 测试失败数量 = 设计的测试用例数量

**验证命令**:
```bash
# 1. 检查测试文件存在
ls -la src/__tests__/week-N/

# 2. 运行测试（必须失败）
npm run test -- week-N

# 3. 统计失败数量
npm run test -- week-N 2>&1 | grep "FAILED"

# 4. 验证测试数量
npm run test -- week-N 2>&1 | grep "Test Files"
```

**验收标准**:
- 测试文件数量 ≥ 1
- 所有测试都是 FAIL 状态（不是 SKIP 或 PASS）
- 失败原因是"功能未实现"，不是测试代码错误

**输出产物**: 测试代码文件（.test.ts）

**QA 签认**: ✅ ___________ (日期)
**PM 验证**: ✅ ___________ (日期)

**⚠️ 关键规则**: 此检查点未通过，DEV 绝对不能开始实现！

---

### 🔴 检查点 3: PM 核准进入实现阶段

**责任人**: PM
**前置条件**: 检查点 2 通过
**触发条件**: QA 完成测试编写

**PM 验证步骤**:
1. [ ] 检查测试文件存在
2. [ ] 运行测试确认 RED 状态
3. [ ] 确认测试失败数量符合预期
4. [ ] 在项目文档中记录"已批准进入实现阶段"

**验证命令**:
```bash
# PM 必须亲自执行
npm run test -- week-N
```

**文档记录**: 在 `docs/week-N-WO.md` 中添加：
```markdown
## 流程状态

- [x] PO 需求定义完成
- [x] QA 测试代码编写完成（RED）
- [x] PM 核准进入实现阶段
- [ ] DEV 实现功能
- [ ] QA 验证测试通过（GREEN）
```

**输出产物**: PM 批准记录

**PM 签认**: ✅ ___________ (日期)

**⚠️ 关键规则**: 只有 PM 签认后，DEV 才能开始实现！

---

### 🔴 检查点 4: DEV 实现功能（GREEN）

**责任人**: DEV
**前置条件**: 检查点 3 通过
**触发条件**: PM 核准进入实现阶段

**DEV 开发纪律**:
- [ ] 只能修改业务逻辑代码
- [ ] 禁止修改测试代码（由 QA 负责）
- [ ] 每次修改后运行测试
- [ ] 目标：让所有测试从 FAIL 变为 PASS

**开发循环**:
```
RED（测试失败）
  ↓
编写业务逻辑
  ↓
GREEN（测试通过）
  ↓
REFACTOR（重构，可选）
  ↓
重复直到所有测试通过
```

**验证命令**:
```bash
# DEV 每次修改后必须执行
npm run test

# 查看测试进度
npm run test -- week-N
```

**验收标准**:
- 所有测试通过（GREEN）
- Week 1-N-1 回归测试通过
- 测试覆盖率 ≥ 70%

**输出产物**: 功能代码 + 测试全绿截图

**DEV 签认**: ✅ ___________ (日期)

---

### 🔴 检查点 5: QA 验证测试通过（GREEN）

**责任人**: QA
**前置条件**: 检查点 4 通过
**触发条件**: DEV 宣布实现完成

**QA 验证步骤**:
1. [ ] 运行 Week N 测试（必须全绿）
2. [ ] 运行 Week 1-N-1 回归测试（必须全绿）
3. [ ] 检查测试覆盖率（≥ 70%）
4. [ ] 如有测试失败，反馈给 DEV 修复
5. [ ] 生成测试报告

**验证命令**:
```bash
# 1. 运行当前 Week 测试
npm run test -- week-N

# 2. 运行全部测试（回归）
npm run test

# 3. 检查覆盖率
npm run test:coverage
```

**验收标准**:
- Week N 测试 100% 通过
- Week 1-N-1 测试 100% 通过
- 测试覆盖率 ≥ 70%

**输出产物**: `docs/week-N-test-report.md`

**QA 签认**: ✅ ___________ (日期)

## 🚫 禁止行为清单

### DEV 禁止做：
- ❌ 在测试代码不存在时开始实现
- ❌ 修改测试代码
- ❌ 提交测试未通过的代码
- ❌ 跳过回归测试

### QA 禁止做：
- ❌ 只写测试用例文档，不写测试代码
- ❌ 修改业务逻辑代码
- ❌ 在测试未全部通过时生成报告

### PM 禁止做：
- ❌ 在未验证测试代码时批准 DEV 开始
- ❌ 跳过任何检查点
- ❌ 忘记更新 CLAUDE.md

## 🔧 故障排除

### 问题 1: 测试代码不存在就尝试实现

**症状**: DEV 开始编码，但没有 .test.ts 文件

**原因**: 跳过了检查点 2

**解决方案**:
1. 立即停止实现
2. 回滚到检查点 1
3. QA 先编写测试代码
4. PM 验证 RED 状态
5. 重新进入检查点 3

### 问题 2: 测试未失败（不是 RED）

**症状**: 运行测试显示 PASS 或 SKIP

**原因**: 测试代码写错了，或者功能已经部分实现

**解决方案**:
1. QA 检查测试代码逻辑
2. 确保测试验证的是"未实现"的功能
3. 如果功能已实现，QA 需要修改测试使其失败
4. 重新验证 RED 状态

### 问题 3: 回归测试失败

**症状**: Week N 测试通过，但 Week 1-N-1 有失败

**原因**: 破坏了冻结的功能

**解决方案**:
1. DEV 立即修复
2. 重新运行所有测试
3. 直到 100% 通过
